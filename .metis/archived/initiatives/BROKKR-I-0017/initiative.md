# Generated Client SDKs for Brokkr API Initiative

## Context

Brokkr already publishes an OpenAPI spec via `utoipa` annotations (`crates/brokkr-broker/src/api/v1/openapi.rs` and every v1 route module). Today there is no published client SDK — internal consumers (notably `brokkr-agent`) hand-roll HTTP calls against the API, and external integrators have no supported path beyond reading the spec.

Generated SDKs would give us a low-cost, multi-language client story driven by a single source of truth. The agent itself can dogfood the generated Rust client, which is the strongest possible forcing function for keeping the spec honest.

This is a deliberate strategic investment: the cost is mostly upfront (generator selection, CI wiring, ergonomic wrapper layer) and ongoing discipline (regeneration in CI, drift detection). The benefit compounds across every future API consumer.

## Goals & Non-Goals

**Goals:**
- Establish the OpenAPI spec emitted by the broker as the single authoritative API contract.
- Ship a Rust SDK and a Python SDK, both generated from that spec, with thin hand-written ergonomic layers (auth via PAK headers, retries, pagination, typed errors).
- Wire CI to regenerate clients and fail the build on undeclared spec drift.
- Dogfood the Rust SDK inside `brokkr-agent` to validate the contract end-to-end.

**Non-Goals:**
- Hand-written SDKs in any language. If a generator can't produce it, we extend the ergonomic wrapper, not the generated core.
- TypeScript / Go / Java SDKs in this initiative. Defer until a real consumer asks; the generators will still be there.
- Changing the broker's HTTP API surface beyond fixes uncovered by the audit (e.g. missing schemas, wrong response types, undocumented endpoints).
- Publishing SDKs to public package registries (crates.io, PyPI). In-repo / internal distribution is sufficient for this initiative.

## Detailed Design

### Source of truth

The broker's `utoipa`-derived OpenAPI document is the contract. The committed artifact lives at `openapi/brokkr-v1.json` and is regenerated by `cargo run -p brokkr-broker --example openapi_export` (added during the audit task).

### Path layout (decision 2026-05-14)

All `#[utoipa::path]` annotations declare paths **without** the `/api/v1/` prefix. The OpenAPI document carries a global `servers: [{ url: "/api/v1" }]` entry so generated clients automatically prepend it. Rationale: keeps annotations focused on resource paths, makes a future v2 prefix cheap, and matches how the runtime router applies the prefix via `.nest("/api/v1", ...)`.

### Error model (decision 2026-05-14)

Every error response uses a single canonical schema:

```
ErrorResponse {
  code: String,         // stable, machine-readable, e.g. "agent_not_found"
  message: String,      // human-readable
  details: Option<Map<String, Value>>,  // optional structured context
}
```

Implementation: define `ErrorResponse` once in the broker, implement Axum `IntoResponse` so handlers can return `Result<T, ErrorResponse>` (or a thin wrapping enum) instead of `(StatusCode, String)`. Reference the schema from every `responses(...)` clause in `#[utoipa::path]` annotations. Status codes stay HTTP-native; `code` is the SDK-friendly key callers pattern-match on.

### Generator choices (proposed, to be confirmed in design phase)

- **Rust:** `progenitor` — produces idiomatic Rust clients with `reqwest`, far cleaner output than `openapi-generator`'s Rust target. Lives in `crates/brokkr-client`.
- **Python:** `openapi-python-client` — modern, type-hinted, `httpx`-based. Lives in `sdks/python`.

One generator per language. No second generators "just in case."

### Ergonomic wrapper layer

Generated clients handle wire-level concerns. A hand-written thin layer on top provides:
- PAK auth header injection
- Retry / backoff policy
- Pagination iterators where relevant
- Typed error surfaces friendlier than raw HTTP status enums

The wrapper is intentionally small. If it grows large, that's a signal the spec is missing something — fix the spec, not the wrapper.

### CI / drift detection

- Job regenerates `openapi/brokkr-v1.json` from the running broker (or from `utoipa` at build time) and fails if the committed copy is stale.
- Job regenerates the SDKs and fails if the committed SDK source diffs.
- Both checks are required on PRs that touch `crates/brokkr-broker/src/api/`.

### Dogfooding

`brokkr-agent`'s broker-facing HTTP code is migrated to the generated Rust SDK. This is the acceptance test for the whole initiative: if the agent can't be cleanly expressed against the SDK, the spec or the wrapper is wrong.

## Alternatives Considered

- **Hand-written SDKs.** Rejected: drift is inevitable, maintenance cost scales linearly with the API surface, and we already have a machine-readable spec.
- **`openapi-generator` for Rust.** Rejected in favor of `progenitor`: openapi-generator's Rust output is functional but not idiomatic, and the generator itself is a heavyweight Java dependency.
- **Spec-first authoring (hand-written OpenAPI, server generated from it).** Rejected: the `utoipa` annotation-driven flow is already in place, works well, and keeps the spec colocated with the handlers that implement it.
- **Skip Python, do Rust only.** Tempting, but Python is the most likely language for ops glue and integration scripts. Adding it now is cheap once the generator pipeline exists.

## Implementation Plan

Decomposition is a human-in-the-loop checkpoint and will happen after this initiative reaches the `design` phase with explicit approval. The first task is already identified:

1. **Audit OpenAPI spec coverage** (first task, scoped before decomposition). Walk every v1 route, every request/response type, every error path, and confirm the `utoipa` annotations match reality. Identify gaps (missing schemas, wrong status codes, undocumented endpoints, untyped error responses) and produce a punch list. This audit gates everything downstream — generators are only as good as the spec.

Subsequent tasks (to be created during decomposition, indicative only):
- Select and prototype Rust generator (`progenitor`) against the audited spec.
- Select and prototype Python generator (`openapi-python-client`).
- Design and implement the ergonomic wrapper layer (Rust first, Python mirroring its shape).
- Wire CI drift detection for spec and generated SDKs.
- Migrate `brokkr-agent`'s broker HTTP code to the generated Rust SDK.
- Documentation: usage examples, auth, error handling, regeneration workflow.

## Status Updates

### 2026-05-14 — Audit findings from BROKKR-T-0130

The spec coverage audit ([[BROKKR-T-0130]]) is complete. The exported `openapi/brokkr-v1.json` contains **42 paths, 61 operations, 50 schemas, 3 security schemes**. The spec has more structural debt than initially expected — it cannot be fed to SDK generators as-is. Full punch list lives in `BROKKR-T-0130`'s status updates; summarized for decomposition planning:

**Must-fix before generation (six items, all blocking):**
- **M1 Path prefix inconsistency** — 13 paths in the spec omit the runtime `/api/v1/` prefix. Generated clients would hit wrong URLs. Fix is mechanical but touches every agents-module annotation plus two health routes.
- **M2 Dangling security schemes** — annotations reference `pak` and `bearer_auth` schemes that aren't registered; only `admin_pak`/`agent_pak`/`generator_pak` exist. Either register the missing schemes or rewrite the route annotations.
- **M3 Duplicate operationIds** — six labels/annotations operationIds collide between agents and templates. Generators will fail or auto-suffix into ugly client APIs. Fix: explicit `operation_id = "..."` per annotation.
- **M4 Untyped error bodies** — every error response declares `application/json` content with no schema; handlers return `(StatusCode, String)`. Needs a canonical `ErrorResponse` type returned uniformly + referenced in `responses(...)`. This is the single biggest ergonomic blocker.
- **M5 Missing routes** — 16+ routes wired in `routes()` builders but not exposed in the spec: entire `webhooks` module (10 routes), agents events/heartbeat/stacks/rotate-pak (5), `generators::rotate_generator_pak`, and stacks labels/annotations (6 routes with no `#[utoipa::path]` annotation at all).
- **M6 Missing schemas** — webhook DTOs (4 in models + 6 in the broker module) entirely absent from `components.schemas`. Falls out of M5.

**Fix-eventually (three items, non-blocking):**
- **F1** Two `agent_events` operations have no `security(...)` clause and inherit nothing.
- **F2** Uneven status-code coverage in `responses(...)` — falls out of M4.
- **F3** `stacks::list_deployment_objects` / `create_deployment_object` may be dead code; confirm before annotating or deleting.

**Accepted-limitation (one item):**
- **A1** Three PAK security schemes all use the same `Authorization` header — broker disambiguates at runtime. SDK ergonomic wrapper hides this. No spec change.

### Implications for decomposition

The first family of tasks must be a **"spec hardening"** group, done in the broker before any generator work begins. Suggested task slices (to confirm at decomposition checkpoint):

1. **Spec hardening: error model.** Introduce `ErrorResponse` schema + Axum `IntoResponse` impl, migrate every handler's error returns, update every `responses(...)` clause. (M4, F2.) Largest single piece of work.
2. **Spec hardening: routes & schemas.** Add missing annotations (stacks labels/annotations, webhooks), register every annotated handler in `ApiDoc.paths(...)`, register webhook schemas in `ApiDoc.components.schemas(...)`. (M5, M6.)
3. **Spec hardening: path prefixes, security, operationIds.** Mechanical sweep across all `#[utoipa::path]` annotations to fix M1, M2, M3, F1.
4. **Spec drift detection (CI).** Wire the `openapi_export` example into a CI job; check committed `openapi/brokkr-v1.json` matches generator output; fail PR if stale. Also handles F3 verification.
5. **Generator selection & prototyping** (Rust + Python). Only after 1–4 land.
6. **Ergonomic wrapper layer** (Rust first).
7. **brokkr-agent migration** to the generated Rust SDK.
8. **Docs.**

Tasks 1–3 are the bulk of the unexpected scope. Tasks 5+ are roughly what the original initiative anticipated.

### Recommended next step

Review the punch list in `BROKKR-T-0130` and decide whether to:
(a) keep this initiative scoped as planned and let spec-hardening dominate the early tasks, or
(b) split spec hardening into a separate, smaller initiative that this one depends on.

Option (b) keeps the SDK initiative's scope clean but adds coordination overhead. Option (a) is simpler. Either way, decomposition into concrete tasks should not start until this is decided. Transition `BROKKR-I-0017` `discovery` → `design` once decided.
